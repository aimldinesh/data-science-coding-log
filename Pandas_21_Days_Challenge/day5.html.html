Why vectorisation matters

Python loops and apply process one row at a time in Python. Vectorised operations process the entire column at once in C — the language pandas is built on. The difference is dramatic.

Computing tax = salary * 0.3 on 1 million rows:

Python for loop ~4,800ms
df.apply(lambda x: x*0.3) ~980ms
df['salary'] * 0.3 ~6ms

Vectorised is ~800x faster than a loop, ~160x faster than apply for this simple case.

# SLOW — Python loop (never do this)
taxes = []
for salary in df['salary']:
    taxes.append(salary * 0.3)
df['tax'] = taxes

# SLOWER than vectorised — apply
df['tax'] = df['salary'].apply(lambda x: x * 0.3)

# FAST — vectorised (always prefer this)
df['tax'] = df['salary'] * 0.3
Interview answer: "Pandas operations are built on NumPy, which executes in compiled C code and processes entire arrays at once. Python loops and apply() call Python bytecode once per row — that interpreter overhead adds up to thousands of function calls. Vectorised ops eliminate those calls entirely."

apply vs map vs applymap — know the difference

This exact question comes up in interviews. Each method has a specific input scope — get the scope wrong and you get a TypeError.

Method Works on Input to function Speed
Series.map() Series only One scalar value at a time medium
Series.apply() Series only One scalar (or can use raw=True for array) medium
DataFrame.apply() DataFrame One column (axis=0) or one row (axis=1) slow
DataFrame.map()* DataFrame One cell at a time (element-wise) slowest

*Called applymap() in pandas <2.1. Renamed to map() in pandas 2.1+.

# Series.map — best for value substitution / lookup
df['gender_code'] = df['gender'].map({'M': 0, 'F': 1})
df['tax']         = df['salary'].map(lambda x: x * 0.3)

# Series.apply — same as map for scalars, but can return Series
df['category'] = df['score'].apply(lambda x: 'high' if x > 80 else 'low')

# DataFrame.apply with axis=1 — receives one row as a Series
df['full_label'] = df.apply(
    lambda row: f"{row['first']} {row['last']} ({row['dept']})",
    axis=1
)

# DataFrame.apply with axis=0 — receives one column as a Series
df.apply(lambda col: col.max() - col.min())  # range per column

# DataFrame.map — element-wise across entire DataFrame
df.map(lambda x: round(x, 2) if isinstance(x, float) else x)
Common mistake: using df.apply(lambda x: x*0.3, axis=1) thinking it applies to each value — but with axis=1, x is an entire row (a Series). The multiplication actually works here by accident, but the intent is wrong and it's much slower than df * 0.3.

Replacing apply with vectorised operations

For every common apply pattern there's a faster vectorised equivalent. Know these cold — interviewers often ask you to optimise slow code.

# ── Conditional column ──────────────────────────────────────

# SLOW
df['level'] = df['salary'].apply(lambda x: 'senior' if x > 80000 else 'junior')

# FAST — np.where for binary condition
df['level'] = np.where(df['salary'] > 80000, 'senior', 'junior')


# ── Multiple conditions ──────────────────────────────────────

# SLOW
def tax_band(x):
    if x < 30000:   return 0.05
    elif x < 60000: return 0.15
    else:            return 0.30
df['tax_rate'] = df['salary'].apply(tax_band)

# FAST — np.select for multiple conditions
conditions = [df['salary'] < 30000, df['salary'] < 60000]
choices    = [0.05, 0.15]
df['tax_rate'] = np.select(conditions, choices, default=0.30)


# ── Value mapping ────────────────────────────────────────────

# SLOW
df['dept_code'] = df['dept'].apply(lambda x: {'Eng':1,'HR':2,'Sales':3}.get(x))

# FAST — .map() with a dict
df['dept_code'] = df['dept'].map({'Eng':1, 'HR':2, 'Sales':3})


# ── String operations ────────────────────────────────────────

# SLOW
df['name_clean'] = df['name'].apply(lambda x: x.strip().lower())

# FAST — str accessor (vectorised C loop)
df['name_clean'] = df['name'].str.strip().str.lower()


# ── Row-wise arithmetic ─────────────────────────────────────

# SLOW
df['profit'] = df.apply(lambda r: r['revenue'] - r['cost'], axis=1)

# FAST — direct column arithmetic
df['profit'] = df['revenue'] - df['cost']
Decision rule: before writing apply, ask — can I express this as arithmetic, np.where, np.select, .map(dict), or a .str accessor? If yes, do that instead. Reserve apply only for logic that genuinely can't be vectorised.

When apply is actually the right tool

apply gets a bad reputation, but there are real cases where it's the correct or only choice. Know them — interviewers respect nuance over blanket rules.

1. Multi-column input, single output

When a new column depends on logic across multiple columns that can't be expressed as simple arithmetic.

# Classify based on two column combination
def risk_level(row):
    if row['age'] > 60 and row['smoker'] == 'yes': return 'high'
    if row['bmi'] > 30:                              return 'medium'
    return 'low'
df['risk'] = df.apply(risk_level, axis=1)

2. Returning multiple new columns

When one function should produce several output columns at once.

def parse_address(addr):
    parts = addr.split(',')
    return pd.Series({'city': parts[0].strip(),
                       'state': parts[1].strip() if len(parts)>1 else None})

df[['city', 'state']] = df['address'].apply(parse_address)

3. Applying a column-level function

Summarise or transform an entire column as a unit (axis=0).

# coefficient of variation per column
df.select_dtypes('number').apply(
    lambda col: col.std() / col.mean()
)

4. Calling external functions / APIs per row

When each row needs to call a function you don't control — a geocoder, a model prediction, an API call. In this case apply is unavoidable. Mitigate with caching and batching.

Interview nuance: "apply is a last resort, not a first choice. I default to vectorised ops and np.where/np.select. I reach for apply when logic involves multiple columns, returns multiple outputs, or calls external functions. I always profile with %%timeit before optimising."

Quick quiz — 4 questions

No peeking. Think through each before clicking.

Q1. What does df.apply(func, axis=1) pass to func?

Correct! axis=1 means "apply across columns" — so func receives each row as a Series, where the index is the column names. axis=0 (default) passes each column as a Series.
axis=1 means the function is applied row-wise. Each call receives one row as a Series (index = column names). axis=0 (default) passes one column at a time.

Q2. What is the fastest way to create a column: 1 if salary > 80000 else 0?

Correct! Boolean comparison on a Series is fully vectorised and returns a boolean Series. astype(int) converts True→1, False→0 — also vectorised. This is the fastest and most idiomatic pandas pattern.
Both apply and map iterate row by row in Python. The fastest approach is (df['salary'] > 80000).astype(int) — the comparison is vectorised in C, and astype is also vectorised. No Python loop at all.

Q3. You need to assign one of three tax rates based on three income brackets. What's the best vectorised approach?

Correct! np.select is designed exactly for multiple condition branches. Nested np.where works but becomes unreadable with 3+ conditions. np.select is cleaner, vectorised, and easier to maintain.
For 3+ conditions, np.select is the right tool. It takes a list of conditions and a matching list of return values, with a default for unmatched rows. Much cleaner than nested np.where and faster than apply.

Q4. What is the difference between Series.map() and Series.apply() for scalar functions?

Correct! For lambda/scalar functions, map and apply on a Series are nearly identical in behaviour and speed. The key difference is that map() additionally accepts a dict (for value substitution) or another Series (index-aligned lookup) — making it more versatile for encoding tasks.
For scalar lambda functions, map() and apply() on a Series behave almost identically. The practical difference is that map() also accepts a dict or Series for lookup/substitution — e.g. df['gender'].map({'M':0,'F':1}) — which is cleaner and slightly faster than a lambda for that use case.

# Setup — 500k row DataFrame
import pandas as pd
import numpy as np

np.random.seed(42)
n = 500_000
df = pd.DataFrame({
    'salary'  : np.random.randint(20000, 150000, n),
    'age'     : np.random.randint(22, 65, n),
    'dept'    : np.random.choice(['Eng','HR','Sales','Finance'], n),
    'score'   : np.random.uniform(0, 100, n),
    'name'    : ['  alice  ', 'BOB', 'Carol '] * (n // 3) + ['dave'] * (n % 3)
})

df['salary'] * 0.3
