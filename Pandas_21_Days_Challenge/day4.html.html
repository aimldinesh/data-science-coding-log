Understanding dtypes

Every column in a DataFrame has a dtype. Wrong dtypes silently break operations — sorting, arithmetic, groupby, merges. Always check dtypes first.

dtypepandas aliasWhen you'll see it
int64int64Integer numbers. Becomes float64 if NaNs are present (NaN is float).
float64float64Decimals, or any int column that had a NaN at load time.
objectobjectStrings (and mixed types). Default for anything pandas can't infer. Slow.
boolboolTrue/False. Watch out — mixed True/False/"True" loads as object.
datetime64[ns]datetime64Timestamps. Must be explicitly parsed — won't auto-convert from strings.
categorycategoryLow-cardinality strings/ints. Huge memory savings. Enables ordered sorts.
Int64Int64 (capital I)Nullable integer (pandas ≥1.0). Allows NaN in integer columns.
# Inspect types
df.dtypes
df.info()                          # types + null counts + memory
df.select_dtypes(include='number') # only numeric columns
df.select_dtypes(include='object') # only string columns
df.select_dtypes(exclude='number') # exclude numeric

# Cast types
df['age']    = df['age'].astype('int64')
df['date']   = pd.to_datetime(df['date'])
df['price']  = pd.to_numeric(df['price'], errors='coerce')  # bad → NaN
df['active'] = df['active'].map({'Y': True, 'N': False})
Classic trap: a column like revenue loads as object because some rows contain "$1,200" with a dollar sign. df['revenue'].sum() will concatenate strings instead of summing numbers. Always cast after stripping non-numeric characters.

The category dtype — massively underused

Converting low-cardinality string columns to category can cut memory by 5–10x and speed up groupby operations. Almost nobody uses it — which makes it a great interview talking point.

Memory usage — 1M row DataFrame with a "country" column (200 unique values):

object dtype~76 MB
category dtype~1.2 MB

category stores integer codes + a lookup table — strings are never repeated in memory

# Convert to category
df['country'] = df['country'].astype('category')
df['grade']   = df['grade'].astype('category')

# Ordered category — enables correct sort and comparison
from pandas.api.types import CategoricalDtype
grade_order = CategoricalDtype(['F','D','C','B','A'], ordered=True)
df['grade'] = df['grade'].astype(grade_order)

df[df['grade'] > 'C']   # works because ordered=True
df.sort_values('grade')  # sorts F → A correctly

# Check category codes and categories
df['country'].cat.codes        # integer codes stored internally
df['country'].cat.categories    # unique values

# Convert all low-cardinality object columns at once
for col in df.select_dtypes('object'):
    if df[col].nunique() / len(df) < 0.5:   # less than 50% unique → category
        df[col] = df[col].astype('category')
Rule of thumb: if a column has fewer than 50% unique values relative to row count, it's a candidate for category. Gender, country, status, department — almost always category.

String cleaning with the str accessor

Real-world data is messy — inconsistent casing, extra whitespace, mixed formats. The .str accessor gives you vectorised string operations — no loops needed.

# Case and whitespace
df['name'].str.lower()
df['name'].str.upper()
df['name'].str.strip()          # remove leading/trailing spaces
df['name'].str.strip().str.lower() # chain them

# Replace and remove
df['phone'].str.replace(r'\D', '', regex=True)     # remove all non-digits
df['price'].str.replace(r'[$,]', '', regex=True)   # remove $ and commas

# Filter / detect
df[df['email'].str.contains('@gmail', na=False)]
df[df['code'].str.startswith('TX')]
df[df['name'].str.match(r'^[A-Z]')]             # starts with capital

# Split and extract
df['full_name'].str.split(' ', expand=True)       # → two columns
df[['first', 'last']] = df['full_name'].str.split(' ', expand=True)
df['area_code'] = df['phone'].str.extract(r'\((\d{3})\)') # regex group

# Length and padding
df['name'].str.len()
df['code'].str.zfill(5)          # pad with zeros: '42' → '00042'

Handling duplicates

Duplicate rows silently skew every aggregate — counts, means, groupby results. Always check before analysis.

# Detect duplicates
df.duplicated().sum()                       # total duplicate rows
df.duplicated(subset=['user_id']).sum()    # duplicated on a key column
df[df.duplicated(keep=False)]             # show ALL copies (not just 2nd+)

# Remove duplicates
df.drop_duplicates()                                    # all columns must match
df.drop_duplicates(subset=['user_id'])             # based on key column
df.drop_duplicates(subset=['user_id'], keep='last') # keep most recent
df.drop_duplicates(keep=False)                      # drop ALL copies of dupes

keep='first' (default)

Keeps the first occurrence of each duplicate group. Good when data is sorted by timestamp — first = earliest record.

keep='last'

Keeps the last occurrence. Good when data is sorted by timestamp — last = most recent update (e.g. updated customer records).

# Real-world pattern: deduplicate events, keep latest per user
df_sorted = df.sort_values('event_time')
df_deduped = df_sorted.drop_duplicates(subset=['user_id'], keep='last')

# Check for near-duplicates (same name, different case)
df['name_clean'] = df['name'].str.lower().str.strip()
df.drop_duplicates(subset=['name_clean'])

# Find duplicate user_ids to investigate
dupe_ids = df[df.duplicated(subset=['user_id'], keep=False)]
dupe_ids.sort_values('user_id')
Interview pattern: when you see duplicates, always ask why before dropping. Are they from multiple data sources being concatenated? A logging bug? Intended (same user, multiple orders)? The reason determines the fix.

Quick quiz — 4 questions

Answer before revealing. These are real interview questions.

Q1. A numeric column loads as object dtype. What's the most likely cause?

Correct! Even one non-numeric value forces the entire column to object dtype. Fix: strip characters then use pd.to_numeric(col, errors='coerce') to convert valid values and turn bad ones into NaN.
NaN values don't force object — they just make integers become float64. The real culprit is one or more non-numeric strings like "$1,200" mixed in. Fix with str.replace then pd.to_numeric(errors='coerce').

Q2. When should you use the category dtype?

Correct! category saves memory by storing integer codes internally with a lookup table. It's most effective when the number of unique values is much smaller than the number of rows — like country, status, gender, department.
category works best for low-cardinality columns — where unique values are a small fraction of total rows. There's no hard threshold like 10, but a rule of thumb is <50% unique values relative to row count. High-cardinality columns (like names, IDs) don't benefit.

Q3. What does df['phone'].str.replace(r'\D', '', regex=True) do?

Correct! \D is the regex pattern for "any non-digit character". With regex=True, this strips everything that isn't 0–9 — parentheses, dashes, spaces, country codes — leaving only the raw digits.
\D in regex means "any character that is NOT a digit". So str.replace(r'\D', '', regex=True) removes all non-digits — perfect for standardising phone numbers like "(91) 98765-43210" → "919876543210".

Q4. You have a customer table with duplicate user_ids because records were updated. How do you keep only the most recent record per user?

Correct! Default keep='first' keeps the earliest record. You need to sort by the timestamp column first so that 'last' means most recent. keep=False would delete all copies of duplicated rows — not what you want.
Without sorting first, keep='first' or keep='last' is arbitrary. Sort by your timestamp column ascending, then drop_duplicates(subset=['user_id'], keep='last') retains the most recent row per user.
