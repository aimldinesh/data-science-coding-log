File formats — know when to use each

Interviews often ask "how would you load this data?" — the right answer depends on the format and size.

CSV

Most common. Human readable. Slow for large files. No type info — everything loads as string by default.

read_csv / to_csv

Excel (.xlsx)

Multiple sheets. Preserves formatting. Slower than CSV. Use when receiving files from non-technical stakeholders.

read_excel / to_excel

JSON

Common from APIs. Can be nested. Use json_normalize for nested structures. Larger file size than CSV.

read_json / to_json

Parquet

Columnar format. Compressed. Fast. Preserves dtypes. Best for large ML datasets. Industry standard in data engineering.

read_parquet / to_parquet

SQL

Read directly from databases. Requires SQLAlchemy connection. Returns a DataFrame just like read_csv.

read_sql / to_sql

Others

read_html (scrape tables), read_clipboard, read_feather, read_hdf — exist but rarely tested in interviews.

Interview tip: if asked "what format would you use to store a 10GB ML dataset?", say Parquet — it's columnar, compressed, fast to read partial columns, and preserves dtypes. CSV loads 5-10x slower for large files.

read_csv — the most important parameters

read_csv has 50+ parameters. These are the ones that come up in interviews and real work.

ParameterExample valueWhat it does
sep'\t' or ';'Delimiter (default comma). Use '\t' for TSV files.
header0 or NoneRow number for column names. None = no header row.
index_col'id' or 0Column to use as the row index.
usecols['a','b','c']Load only these columns — saves memory on large files.
dtype{'id':'str'}Force column types on load — prevents wrong inference.
parse_dates['date'] or TrueParse these columns as datetime automatically.
na_values['N/A','unknown']Extra strings to treat as NaN beyond the defaults.
nrows1000Load only first N rows — great for exploring big files.
chunksize10000Return an iterator of chunks instead of loading all at once.
encoding'utf-8' or 'latin1'File encoding. Use 'latin1' when utf-8 fails on older files.
skiprows2 or [0,2]Skip N rows from top, or skip specific row numbers.
# Minimal — just load it
df = pd.read_csv('data.csv')

# Production-grade — explicit and safe
df = pd.read_csv(
    'data.csv',
    usecols   = ['id', 'date', 'amount'],     # load only needed columns
    dtype     = {'id': 'str'},                # don't auto-infer id as int
    parse_dates = ['date'],                   # parse as datetime
    na_values = ['N/A', 'unknown', '-']        # extra null markers
)
Classic bug: loading a ZIP code like "07030" without dtype={'zip':'str'} — pandas infers it as int64 and silently drops the leading zero. Always force string types for IDs, zip codes, phone numbers.

Excel, JSON, Parquet & SQL

Each format has its own quirks. Here's what you need to know for each.

# Excel — multiple sheets
df   = pd.read_excel('report.xlsx')                    # first sheet
df   = pd.read_excel('report.xlsx', sheet_name='Sales') # by name
all_sheets = pd.read_excel('report.xlsx', sheet_name=None) # dict of dfs
# JSON — flat and nested
df = pd.read_json('data.json')

# Nested JSON from an API — use json_normalize
import json
with open('api_response.json') as f:
    data = json.load(f)
df = pd.json_normalize(data['results'], sep='_')  # flattens nested keys
# Parquet — fast, typed, compressed
df.to_parquet('data.parquet')
df = pd.read_parquet('data.parquet')
df = pd.read_parquet('data.parquet', columns=['id','amount'])  # columnar — only read what you need
# SQL — requires SQLAlchemy
from sqlalchemy import create_engine
engine = create_engine('postgresql://user:pass@host/dbname')

df = pd.read_sql('SELECT * FROM orders WHERE status = \'paid\'', engine)
df = pd.read_sql_table('orders', engine)
df.to_sql('orders_clean', engine, if_exists='replace', index=False)

Writing — key options

df.to_csv('out.csv', index=False)
df.to_excel('out.xlsx',
    sheet_name='Results',
    index=False)
df.to_parquet('out.parquet',
    compression='snappy')

Common mistake

Forgetting index=False when writing CSV — adds a useless numbered index column that then confuses the next person who loads the file.

performance

Chunking large files

What do you do when the file is 10GB and your RAM is 16GB? You don't load it all at once — you chunk it.

How chunking works — file is processed in pieces:

File on disk
rows 0–9999
rows 10k–19999
rows 20k–29999
rows 30k–39999
↓
In memory
process chunk → aggregate → discard
# chunksize returns a TextFileReader iterator
chunks = pd.read_csv('big_file.csv', chunksize=10000)

# Pattern 1 — filter and collect matching rows
result = []
for chunk in chunks:
    filtered = chunk[chunk['status'] == 'paid']
    result.append(filtered)
df = pd.concat(result, ignore_index=True)

# Pattern 2 — aggregate (e.g. sum revenue per category)
totals = {}
for chunk in pd.read_csv('big_file.csv', chunksize=10000):
    grouped = chunk.groupby('category')['revenue'].sum()
    for cat, val in grouped.items():
        totals[cat] = totals.get(cat, 0) + val

# Pattern 3 — quick exploration without loading all
df_sample = pd.read_csv('big_file.csv', nrows=1000)  # inspect first 1000 rows
Interview answer: when asked "how do you handle a file that doesn't fit in memory?" — say chunking with chunksize, or read only needed columns with usecols, or use Parquet with columnar reads. Bonus: mention Dask or PySpark for truly large-scale work.

Quick quiz — 3 questions

No peeking. Answer from memory.

Q1. You load a CSV with a column "user_id" containing values like "00123". After loading, all values become 123 (no leading zero). What caused this and how do you fix it?

Correct! Pandas auto-infers numeric types and drops leading zeros. Force the column to string with dtype={'user_id': 'str'} at load time.
The issue is type inference — pandas reads it as an integer and drops the leading zero. Fix: dtype={'user_id': 'str'} forces it to stay as a string.

Q2. Why is Parquet preferred over CSV for large ML datasets?

Correct! Parquet is columnar (read only the columns you need), compressed (smaller file), preserves dtypes (no re-casting on load), and significantly faster than CSV at scale.
Parquet's advantages: columnar storage (read only the columns you need), compressed by default, preserves dtypes (no type inference), and much faster I/O than CSV for large files.

Q3. What does pd.read_csv('file.csv', chunksize=5000) return?

Correct! chunksize returns an iterator — each iteration gives you a DataFrame of up to chunksize rows. Nothing is fully loaded into memory until you iterate.
chunksize returns a TextFileReader iterator, not a list or a single DataFrame. You loop over it — each loop gives you one chunk (up to 5000 rows) without loading the whole file.
